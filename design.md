# broonie — Design Document

## Overview

A self-hosted server application that watches GitHub repositories for issues and autonomously implements them using an AI agent pipeline. The user jams with an AI to break a feature into well-structured GitHub issues, goes to bed, and wakes up to pull requests.

The system is intentionally split into two concerns:

- **Go app** — handles all deterministic lifecycle work: polling, scheduling, git, branches, worktrees, PRs, GitHub state, web GUI
- **AI agent** — handles only the coding: Blip (inside pi, inside nono) implements the issue, verifies its work, and signals completion

---

## Issue Format

Issues carry machine-readable metadata in a collapsed `<details>` block at the top of the body. Humans don't see it by default; the poller finds it by scanning for the block.

````markdown
<details>
<summary>Metadata</summary>

```json
{
  "type": "AUTO",
  "depends-on": ["#12", "#15"]
}
```

</details>

Normal issue body follows here.
````

- **`type: AUTO`** — the agent pipeline picks this up and implements it
- **`type: HITL`** — Human in the Loop; the system skips this issue until a human acts on it
- **`depends-on`** — list of issue numbers that must be closed before this one is worked on
- Unknown JSON keys are ignored (forward-compatible)

A server-side validator (triggered on each poll cycle) checks metadata validity. Malformed issues are labeled `invalid-frontmatter`.

### Issue Creation

Issues are created by an AI using a dedicated skill (`skills/create-issue/`). The skill defines exactly how to produce valid frontmatter and how to split work into right-sized issues. What happens before invoking the skill (interview, jam session, one-shot prompt) is up to the user.

## Agent Pipeline (per issue)

The Go app owns all lifecycle steps. The AI owns only the coding.

```
1. git worktree add ./workspaces/<issue-number> -b <branch-name>
      └─ branch name generated by a fast/cheap model from the issue title

2. Loop up to N iterations:
      → build prompt from issue title + body
      → drive nono through its Go SDK (`nono-go`) — never the `nono` CLI —
        launching pi inside a sandbox built from the custom nono profile
        authored in this repo (replaces `always-further/pi`):
            pi --print --approve --agent blip \
               --session-id <issue-number> --continue "$(cat prompt.md)"
      → pi runs in the sandbox; stdout is captured to the session log only — it is never parsed for control flow
      → each iteration, poll the worktree `.loop/` directory for the agent's handoff signal
      → if `.loop/done` and `.loop/pr.md` are both present: proceed to step 3
      → otherwise: sleep, next iteration

3. On `.loop/done` present:
      → read `.loop/pr.md`
      → git push
      → open PR using `.loop/pr.md` as the body
      → mark issue done in SQLite
      → git worktree remove

4. If max iterations reached without `.loop/done`:
      → label issue `stalled` on GitHub
      → mark stalled in SQLite
      → git worktree remove
```

### Handoff Signal (`.loop/`)

The agent communicates completion to the Go app **only** through files in a `.loop/` directory at the worktree root. This is the sole control-plane channel between the sandboxed agent and the Go app.

- `.loop/pr.md` — the pull request description
- `.loop/done` — presence signals the run is complete and verified

Completion is defined as **both** files existing. The Go app never parses pi's stdout for control flow — stdout is logs-only (served via pi session `.jsonl`). A file-presence check is boolean and cannot false-positive the way a token scan can (the token could appear mid-run in a quote, a comment, or a hallucinated "I will now output..."). Blip writes `.loop/done` as its final step, after verification.

> Future: `.loop/stalled` for the agent to signal an unimplementable issue explicitly, rather than burning all iterations.

### Prompt Template

Kept minimal — Blip handles codebase exploration, planning, and verification internally.

```
You are working on the following GitHub issue in this repository.
Implement it completely and verify your work.

When done:
1. Write a pull request description to `.loop/pr.md`
2. As your final action, after verifying your work, create an empty file at `.loop/done` to signal completion

---
Title: {{issue.title}}
Body: {{issue.body}}
```

### Agent Stack

| Layer | Tool |
|---|---|
| Sandbox | nono, driven via the `nono-go` Go SDK — custom profile authored in this repo (no `nono` CLI exec) |
| Harness | pi (`--print --approve --agent blip --session-id <n> --continue`) |
| Agent | Custom Blip implementation (forked from Blip, lives in this repo) — writes `.loop/pr.md` and signals completion via `.loop/done`, tuned for unattended autonomous operation |
| Session memory | pi session store (`PI_CODING_AGENT_SESSION_DIR` per repo) |

Blip's inner pipeline replaces the need for a naive retry loop. It already surveys the codebase, plans, implements, self-reviews, and verifies before declaring success.

## Scheduling

- Poll each watched repo every N minutes (configurable per repo)
- On each poll: fetch open AUTO issues, resolve dependency graph, pick next unblocked issue
- **Serial execution** — one issue at a time per repo (parallel in a future version)
- No webhooks required for v1

## Go Application

### Stack

| Layer | Tool |
|---|---|
| Language | Go |
| HTTP | `net/http` |
| Frontend | Go `html/template` + HTMX + Pico CSS |
| Database | SQLite (`modernc.org/sqlite`) |
| GitHub API | `google/go-github` |
| nono | `github.com/always-further/nono-go` SDK (native Go, no CLI exec) |
| Container | Single binary, slim Docker image |

### Database (SQLite)

Stores repo configs, issue queue state, and run metadata.

```
repos         — watched repos, per-repo config (encrypted secrets)
issues        — mirrored issue state, status, depends-on, type
runs          — per-issue run history (iterations, outcome, timestamps)
```

### Secret Storage

Sensitive fields (GitHub tokens, SSH keys) are encrypted at rest using a `SECRET_KEY` environment variable passed into the container.

> TODO: harden secret storage in a future version (external secrets manager, etc.)

## Web GUI (v1 — lean)

- **Repo list** — add/remove watched repos
- **Per-repo config** — GitHub token, SSH key, poll interval, pi agent config, nono profile
- **Per-repo queue** — issues with status: pending / blocked / in-progress / done / stalled

Deferred to future versions:
- Run logs (pi session output, nono audit log)
- Parallel execution controls
- Priority / advanced scheduling
- Full settings UI

> Run logs will be served by the Go API reading pi session `.jsonl` files from a mounted volume. No direct filesystem access required for the user.

---

## Docker

Single container:
- Go binary (web server + poller + worker)
- SQLite database (volume-mounted for persistence)
- pi session directory (volume-mounted for run log access)
- Git repos / worktrees (volume-mounted)

Environment variables:
- `SECRET_KEY` — encryption key for stored secrets
- `PI_CODING_AGENT_SESSION_DIR` — pi session directory (per-repo isolation)

---

## Guiding Principles

- **The Go app is dumb and reliable** — it never makes judgment calls, only mechanics
- **The AI is isolated and sandboxed** — nono ensures it can only touch the worktree
- **Serial first, parallel later** — keep v1 simple, use the system to improve the system
- **Lean GUI, lean schema** — ship something that works, iterate with the tool itself
